## 🟡 検討すべきトレードオフ

- [ ] #003 **DP 接続では 32U720A-B の USB ハブ機能が使えない可能性があるかもしれません**
  32U720A-B の USB-C 端子は「USB ハブ＋映像入力＋PD 90W 給電」を兼ねた upstream ポートです。今回の構成では Anker Prime DP 2.1 → 32U720A-B の DP 1.4 入力を使うため、モニター内蔵 USB ハブ（USB-A ポート等）は Mac Mini から使えない可能性があります。Mac Mini にはすでに十分な USB ポートがあり Anker Prime 経由でも接続できるため実害は少ないかもしれませんが、把握しておく価値があります。
  🟡 リスク: 低（Mac Mini 本体や Anker Prime の USB ポートで代替可能）
  > 返答:

- [ ] #004 **Windows WS 側のモニター接続ポートの割り当てが未整理かもしれません**
  接続図は Mac Mini → Anker Prime → DP+HDMI の部分のみ更新されています。Windows WS が各モニターのどのポートを使うかが具体的でないかもしれません。
  参考として考えられる構成：
  - 32U720A-B: DP 1.4 は Mac（Anker Prime 経由）使用 → Windows は **HDMI 2.1 または USB-C** 入力を使う
  - 32UL750-W: HDMI port 1 は Mac（Anker Prime 経由）使用 → Windows は **HDMI port 2 または DP 1.4** 入力を使う
  両モニターとも複数入力があるため接続自体は可能ですが、WS 購入時に GPU の出力端子と照合が必要です。

## ❓ 確認すべき前提・将来への影響

- [ ] #005 **Mac 使用中に Windows を起動すると AID が誤検知する可能性があるかもしれません**
  Mac Mini 使用中、Anker Prime は常時接続されて DP・HDMI 信号を両モニターに送り続けています。この状態で Windows WS の電源を入れると、Windows の映像信号発生を AID が検出して **Mac 作業中に突然モニターが Windows に切り替わる** 可能性があります。32U720A-B と 32UL750-W の AID 設定を「最終入力優先」または「手動」に変更することで回避できますが、購入後すぐに設定の確認・調整が必要かもしれません。
  > 返答:

- [ ] #006 **Anker Prime の TB5 upstream を TB4 の Mac Mini で使う場合の帯域が未確認かもしれません**
  TB5 ドックを TB4 ホスト（Mac Mini M4）に接続すると TB4 速度（最大40Gbps）に制限されます。4K@60Hz デュアルディスプレイ + 通常の USB 周辺機器の合計帯域は TB4 の40Gbps で十分と考えられますが、多数の高速デバイスを同時接続した場合の帯域圧迫は理論上起こりえます。現在の使用目的では問題にならない可能性が高いです。
  > 返答:

## 🔁 DisplayLink 再検討（2026-06-25 追記分）

- [ ] #013 **🔴 HDCP制限は「DisplayLinkモニター以外」にも波及する可能性が高い**
  idea §「DisplayLink再検討の動機」で「非DisplayLinkモニターは影響を受けないなら運用回避できる」と期待しているが、これは外れる可能性が高い。
  macOS の HDCP 実装では「DisplayLink 接続デバイスがシステムに存在する」だけで、直接接続のモニター（非 DisplayLink）でも Netflix・Apple TV+ の再生が止まるという報告が多数ある。Apple 側の仕様として「HDCP 非準拠デバイスがある場合は全出力を保護」となっているため、「Monitor 1（直接接続）で Netflix を見る」という回避策が機能しない可能性がある。
  **このリスクが事実なら、DisplayLink の除外判断は覆らない**（最も大きな障壁）。
  > 返答:

- [ ] #014 **❓ Mac Mini でのスリープ問題は「ゼロにはならない」可能性がある**
  MacBook のスリープ復帰問題は、スリープ・ウェイクのサイクルが原因のため「Mac Mini = 常時起動」運用ではリスクが下がるのは正しい。ただし DisplayLink の問題は「スリープ」だけに起因しない：
  - macOS アップデートのたびに DisplayLink Manager との相性問題が再発する（1〜2週間の動作不可期間が生じることがある）
  - スリープを完全に無効化しても、Mac Mini が省エネのため一部 sleep like 状態になることがある
  スリープ問題の「頻度」は下がるが「ゼロ」にはならず、macOS アップデートリスクは Mac Mini でも同等に存在する。
  > 返答:

- [ ] #015 **❓ A83B35A1 の Mac Desktop 使用報告は少なく「問題なし」と断言しにくい**
  A83B35A1（Anker Prime A83B35A1）は比較的新しいモデルであり、Mac Mini での使用報告は MacBook に比べてほとんど存在しない。DisplayLink の問題は macOS ドライバーレベルの問題のため、Mac Mini/MacBook に関わらず同じドライバーを使う以上、根本的リスクは変わらない。「Mac Desktop だから安心」というエビデンスは現状存在しない。
  > 返答:

- [ ] #016 **💭 DisplayLink の摩擦は「慣れ」で解決しない種類の問題が多い**
  ターミナル・エディタ・ブラウザ用途での画質差・遅延は実用上ほぼ気にならない（その意味では「慣れたら気にならない」）。しかし下記は慣れで解決しない：
  - DRM 再生不可の問題は毎回「あ、DisplayLink のせいか」という想起が発生する認知コスト
  - macOS アップデート後に「動かなくなる」体験は作業を中断させ、対処に時間がかかる
  - DisplayLink Manager というプロセスが常に走っていることへの心理的負荷
  しょうごさんの「デスクをスッキリさせたい・継続的な摩擦を嫌う」というスタイルと、DisplayLink の「見えない摩擦」は相性が悪い可能性がある。
  > 返答:
