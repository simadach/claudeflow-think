## 🔴 重大なリスク

- [ ] #001 **「AIがカードを自動配置する」体験は、最初の1〜2回は驚きを与えるが飽きる可能性がある**  
  Notion AI・Figma AI・Perplexityなど「AIが何かを自動生成する」UIは数多くあり、ユーザーが「AIに任せると思った通りにならない」と感じた瞬間に手動操作を好むようになります。「AIが配置する」ことへの驚きが薄れた後に残る体験価値は何か——それが説明できないと、このプロダクトは最初の印象だけで終わるかもしれません。
  > 返答:

- [ ] #002 **ブラウザSPAでのWeb検索・画像取得はCORS・クローリング問題でフェーズ1から詰まる可能性がある**  
  「Tesla Model 3を調べて → 画像・Web記事が自動配置」を実現するには、バックエンドサーバー（プロキシ・クローラー）が不可欠です。ブラウザのみでサードパーティサイトの画像・HTMLを取得することは現代のCORSポリシー上ほぼ不可能です。「フェーズ1はローカルだけでいい」という方針でも、バックエンドなしでは最初のユースケースが動きません。フェーズ1の実現コストが想定より大きくなる可能性はないでしょうか。

## 🟡 トレードオフ・代替案

- [ ] #003 **Notion・Obsidian Canvas・Miro・Figma はすでに「カードベースの情報空間」を持っている**  
  差別化は「AIが自動配置する」点ですが、Notion AI・Obsidian Canvas + Claude MCP・Figma AI などは既にこの方向に進んでいます。「新しいカテゴリを作る」という野心は魅力的ですが、既存ツールに MCP/Plugin を追加するアプローチと比較したとき、一から作ることのコスト対効果は十分に検討されていますか？
  > 返答:

- [ ] #004 **「個人ツール → 他人向けプロダクト」移行時の最大の設計負債はデータ永続化**  
  フェーズ1で「認証・スケーラビリティは後回し」としているのはフェーズ1の判断として正しいですが、Workspace の状態（カードの配置・内容）をどこに保存するかの設計が後回しになると、フェーズ2でのリアーキテクチャが大きくなります。ローカルストレージ、IndexedDB、クラウドDB——永続化レイヤーの選択は最初から疎結合にしておかないと移行コストが高くなるかもしれません。
  > 返答:

- [ ] #005 **Claude API の function calling ではなく Claude Code の MCP という選択肢**  
  Workspace API を「Claude がツールとして呼び出す」設計は function calling / tool use で実現できますが、すでに ClaudeFlow や Claude Code を使いこなしているなら、Workspace を MCP サーバーとして実装し Claude Code から制御するアーキテクチャのほうが、既存の自動化インフラとの統合が自然かもしれません。Web UIが不要になる場面もあるという前提で、検討されましたか？
  > 返答:

- [ ] #006 **カードロードマップの10種類は多すぎる可能性がある**  
  Image→Web→Markdown→Video→PDF→Note→HA→Three.js→Browser→Musicの順序は論理的ですが、個人開発のサイドプロジェクトで10種類のカードを実装し切れる見通しはありますか？最初の3種（Image・Web・Markdown）で「AIが情報空間を編集する」というコア体験が検証できるなら、それ以降は検証後に追加するほうがリスクが低いかもしれません。

## ❓ 確認すべき前提

- [ ] #007 **「AIがカードの位置・サイズを決める」具体的な実装イメージはあるか**  
  `createCard(type, content, position)` のような API があるとして、Claudeにどのような座標・サイズを返させるつもりでしょうか。画面サイズ・既存カードの位置・ユーザーの過去の好みなど、レイアウトの最適化は実装してみると想定より複雑な問題になるかもしれません。
  > 返答:

- [ ] #008 **Home Assistant・HEMS・Teslaとの統合はフェーズ何を想定しているか**  
  動機の1つは「HA/TeslaMate/HEMSを一つのワークスペースで束ねる」ですが、カードロードマップではHome Assistant Cardはフェーズ7です。自分自身にとっての最大の価値が後回しになっていて、モチベーションが維持できるか懸念はないでしょうか。
  > 返答:

- [ ] #009 **「検索エンジンの次の世界観」という野心と、サイドプロジェクトの現実のギャップ**  
  「AIネイティブなデスクトップ環境という新しいカテゴリを作る」という表現は非常に大きなビジョンです。一人・サイドプロジェクト・フルタイムではないという制約の中で、このビジョンに向かって「手を動かし続けられるか」を最初に確認する必要はないでしょうか。小さな成功体験（最小プロトタイプの動作）が先に見えているのは良い設計ですが。

  カード配置・ドラッグ&ドロップ・空間的なレイアウトはデスクトップ体験として設計されていますが、日常の情報収集（Tesla を調べる・HA を確認する）はスマートフォンで行う場面も多いのではないでしょうか。将来の拡張に「Vision Proのような空間UI」があり、ビジョンの中に縦長小画面が見当たらないことが少し気になります。
  > 返答:

- [ ] #011 **自分が使い続けるためのモチベーション設計があるか**  
  他のユーザーへの価値が語られていますが、しょうご自身が「毎日使いたい」と思えるユースケースは何でしょうか。Home Assistant Cardがフェーズ7に置かれているということは、自分の日常での使用開始はかなり先になります。「自分が使いながら改善する」フィードバックループが遅くなるリスクはないでしょうか。
  > 返答:

- [ ] #012 **「AIが配置を間違えた時のUX」が設計されていない**
