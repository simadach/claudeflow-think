## 🔴 重大なリスク

- [x] #001 ✅ 反映済み **Nest Mini はカスタム音声エンドポイントとして使えない可能性が高い**  
  Google Nest Mini の音声認識・合成は Google のサーバーに閉じており、サードパーティが「音声I/Oだけ借りる」APIは現時点で公開されていません。Google Home SDK / Cast API でメディア再生（TTS再生）は可能ですが、「Nest Mini に話しかけた音声をカスタムモデルに送る」経路は非公式な回避策（ローカルネットワーク盗聴等）を除き存在しないかもしれません。  
  → **「Nest Mini を音声I/Oとして使う」という核心前提が崩れると、アーキテクチャ全体が成立しない懸念はないでしょうか。**  
  **補記**: HA の `google_generative_ai_conversation` 統合（homeassistant gemini integration）を使う代替案を検討中。この場合、Nest Mini を「出力スピーカー」として使い（Cast API でTTS再生）、音声入力は Assist パイプライン経由で別デバイスまたは HA Companion アプリから行う構成が現実的。Nest Mini をカスタムSTTエンドポイントとして使う前提は回避できる可能性がある。
  > 返答:homeassistant gemini integrationは使えないか？

- [x] #002 ✅ 反映済み **HA の `google_generative_ai_conversation` 統合は「会話アシスタント」として動作するが、プロアクティブ通知には使えない**  
  この統合は HA のアシスタントAPIに接続し、ユーザーが話しかけた時の応答エンジンとして機能します。Gemini 側から「今話しかける」という能動的な発信を HA 経由でトリガーする仕組みは、統合自体には含まれていません。プロアクティブ通知（#3の体験）は別経路（Node-RED・Python スクリプト・HA オートメーション + TTS）が必要になります。  
  → **優先順位3のプロアクティブ提案は、統合だけでは実現できない別設計が必要という認識はあるでしょうか。**（返答確認済み：プロアクティブ通知は後段フェーズに延期。まずユーザーが話しかける方向から実装を進める方針に決定）
  > 返答:まずは私に話しかけることは後段にする。

## ❓ 確認すべき前提

- [x] #006 ✅ 反映済み **Nest Mini は何台あり、どの部屋に置いているか**  
  プロアクティブ提案を「話しかける」場合、どの部屋で誰に向けて話すかの設計が必要です。1台だけなら「洗濯機は別の部屋なのに声が届かない」問題が生じる可能性があります。  
  **補記**: 書斎に1台のみ。まず書斎からスタートする方針が確定。スコープが明確になったことで、初期実装は「書斎での音声インターフェース」に集中できる。将来的な部屋拡張は別フェーズ。
  > 返答:書斎に１台のみ最初はそこからスタート

- [x] #007 ✅ 対象外（返答: 基本的に自分だけ使う。家族・子供が意図せず使うリスクは現時点では低い） **家族（特に子供）が意図せずトリガーする懸念**  
  Gemini が能動的に話しかけるシステムを稼働させると、子供が声を聞いて混乱する・誤認識で家電が操作されるリスクがあるかもしれません。「AI が家の中で話す」体験が家族全員にとって心地よいかどうか、確認・調整の余地はあるでしょうか。
  > 返答:基本私しか使わない

- [x] #008 ✅ 反映済み **Gemini API のコストを「会話のたびに」と捉えた場合の現実的な試算**  
  プロアクティブ提案を1日10回実行すると仮定した場合、月あたりの API コストはどの程度になるか試算されていますか？Gemini Flash（低コスト）と Gemini Pro（高品質）の選択も含め、許容できるコストの上限を先に決めておくと設計の方向が定まりやすいかもしれません。（返答確認済み：事前試算より実使用で判断する経験主義的アプローチ。まず Gemini Flash で始めてコストを観察する形が合っている）
  > 返答:使ってみて判断する

- [x] #009 ✅ 反映済み ✅ 対象外（返答: プライバシーを気にしない。HA データを積極的に Gemini に送る方針） **HA の `sensor` データを Gemini に送る際のプライバシーライン**
